# 完整实验运行流程

本项目的实验流程分为单元测试、正式实验执行、结果查看与结果汇总四个阶段，整体目标是验证框架能否稳定完成 fuzzing，并比较不同调度策略在不同样例上的表现。

首先，在实验开始前，我们先运行单元测试，确认变异器、调度策略、结果输出和汇总脚本均能正常工作。使用的命令如下：

```bash
python -m unittest discover -s simple_fuzzer/tests -p "test_*.py"
```

单元测试通过后，再进行正式实验，避免由于基础模块错误影响最终结果。

其次，在正式实验阶段，我们使用统一实验编排脚本 `run_experiment_suite.py` 组织完整实验。实际使用的命令如下：

```bash
python run_experiment_suite.py --label official --samples 1 2 3 4 --schedules uniform path rare --run-time 10 --repeats 3 --capture-single-runs --save-population
```

本次实验参数设计如下：

- `samples 1 2 3 4`
  覆盖全部 4 个被测样例，保证实验结果具有完整性。
- `schedules uniform path rare`
  其中 `uniform` 作为基线策略，`path` 表示基于路径频率的调度策略，`rare` 表示新增调度策略，用于比较不同调度方法的效果。
- `run-time 10`
  表示每组实验运行 10 秒，使每种策略在同样时间预算下进行比较。
- `repeats 3`
  表示每个“样例-策略”组合重复运行 3 次，并对结果取平均值，以减少单次随机波动带来的影响。
- `capture-single-runs`
  表示额外保存每个样例和策略的一组代表性单次运行结果，便于后续截图和结果查看。
- `save-population`
  表示保存代表性运行结束后的 `population` 快照，用于支撑持久化与中间状态保存相关分析。

整个正式实验共包含：

- 4 个样例
- 3 种策略
- 每组重复 3 次

因此总实验次数为：

```text
4 × 3 × 3 = 36 次
```

再次，在实验执行过程中，脚本会自动记录实验计划，并在 `_result/runs/official/` 目录下生成完整实验产物，包括：

- `experiment_plan.json`
  保存本次实验的参数配置与实验矩阵。
- `comparison/`
  保存批量实验的原始结果和汇总结果。
- `single/`
  保存代表性单次运行结果，包括 `.pkl`、`.json`、`.txt` 以及可选的 `population` 快照。
- `report_assets/`
  保存整理后的汇总表和 Markdown 摘要，便于后续直接写入实验报告或测试报告。

然后，在结果查看阶段，我们使用 `view_result.py` 查看单次运行结果，确认结果文件可以被正确读取。例如：

```bash
python view_result.py _result/runs/official/single/sample-1-path.json
```

该脚本能够输出单次运行的覆盖行数、崩溃数量、路径数、执行次数和运行时长等信息，便于快速检查某组实验的具体表现。

最后，在结果汇总阶段，我们主要使用 `comparison/summary.csv` 和 `report_assets/summary.md` 中的结果，对不同样例和不同调度策略进行比较分析。重点关注的指标包括：

- `avg_execs`
  表示平均执行次数，可反映单位时间内的运行效率。
- `avg_paths`
  表示平均发现的唯一路径数量，可反映路径探索能力。
- `avg_covered_lines`
  表示平均覆盖行数，可反映总体覆盖效果。
- `avg_crashes`
  表示平均崩溃数量，可反映异常触发能力。

总体而言，本实验流程按照“先测基础模块，再跑正式实验，随后查看结果并统一汇总”的顺序展开，既保证了实验过程的稳定性，也保证了结果具有较好的可复现性和可分析性。

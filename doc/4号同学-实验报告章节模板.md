# 4号同学-实验报告章节模板

## 使用说明

这份模板用于填写实验报告中更适合由 4 号同学负责统稿的部分，主要包括：

- 小组分工
- 4.3 持久化与框架优化
- 4.4 入口与运行方式
- 五、测试与结果
- 六、总结与反思中的整体实验组织与结果分析

建议写法：

- 先把命令、参数、结果表格填进去。
- 再补截图、简短分析。
- 最后统一语气和格式。

## 一、小组分工模板

本组围绕灰盒模糊测试框架的关键能力进行了协作开发，分工如下：

- 1 号同学：负责 `mutator.py` 中变异器的实现、补强与测试，重点完成多种输入变异策略及边界情况验证。
- 2 号同学：负责路径频率调度逻辑，实现 `path_power_schedule.py` 与 `path_grey_box_fuzzer.py` 中基于路径频率的 seed 调度。
- 3 号同学：负责新增调度策略的设计、实现与对比实验，完成 `schedule` 包中的扩展策略并与默认策略进行比较。
- 4 号同学：负责结果持久化、实验编排、统一运行入口、结果汇总与实验报告整合，组织整体实验并整理最终统计结果。

如果需要更正式一点，可以改成：

本组采用“核心模块分工 + 实验整合统稿”的协作方式。前 3 位同学分别负责变异器、路径调度和新增调度策略实现，4 号同学负责项目联调、实验编排、结果持久化、结果汇总和实验报告主稿整理，以保证最终提交版本具备可运行、可复现、可分析的特性。

## 二、4.3 持久化与框架优化模板

### 4.3 持久化与框架优化

为了满足实验说明中“使用 `object_utils` 将 seed 或相关中间结果持久化到文件系统，避免长时间运行时内存持续增长”的要求，我们对项目的结果持久化与实验支撑层进行了统一整理。

#### 1. 持久化设计

项目使用 `simple_fuzzer/utils/object_utils.py` 中的工具函数对实验对象进行持久化，主要包括以下几类内容：

- 单次运行结果：
  - 以 `.pkl` 保存 `Result` 对象，便于后续程序化读取。
  - 以 `.json` 保存结构化结果，便于直接生成表格或写入报告。
  - 以 `.txt` 保存简要文本摘要，便于快速查看。
- 中间统计结果：
  - 在需要时保存最终 `population` 快照，用于分析高价值 seed 的分布情况。
- 批量实验结果：
  - 保存原始运行记录 `raw_results.json/csv`。
  - 保存聚合后的 `summary.csv/txt`。
- 报告素材：
  - 自动汇总到 `report_assets/` 目录，方便后续写报告和制作测试 PDF。

#### 2. 输出目录规范

为了避免结果文件命名混乱，我们将输出目录统一设计为：

```text
_result/
├─ single/                # 单次运行结果
├─ comparison/            # 批量对比实验结果
├─ report_assets/         # 汇总后的报告素材
└─ runs/<label>/          # 正式实验套件结果
```

如果使用实验编排脚本，则会在 `_result/runs/<label>/` 下进一步生成：

```text
runs/<label>/
├─ experiment_plan.json
├─ single/
├─ comparison/
└─ report_assets/
```

这样的设计有两个好处：

- 不同实验轮次不会互相覆盖。
- 后续写实验报告时可以直接从统一目录中提取数据和截图。

#### 3. 框架优化内容

除持久化外，我们还对实验支撑层做了以下优化：

- 统一了单次运行与批量运行的结果字段，包括：
  - `sample`
  - `schedule`
  - `run_time`
  - `initial_seed_count`
  - `covered_line_count`
  - `crash_count`
  - `total_paths`
  - `total_execs`
  - `duration_seconds`
- 统一了结果查看方式，`view_result.py` 同时支持 `.pkl` 与 `.json`。
- 新增结果汇总脚本，可自动生成报告所需的表格和 Markdown 摘要。
- 新增正式实验编排脚本，能够自动记录实验计划、实验矩阵和最终产物。

#### 4. 效果说明

通过上述整理，项目从“能跑出结果”提升为“能稳定落盘、能批量汇总、能直接支撑实验分析”的状态。这样不仅减轻了长时间 fuzzing 时的人工整理压力，也提高了结果复现与报告撰写效率。

## 三、4.4 入口与运行方式模板

### 4.4 入口与运行方式

本项目围绕 3 类入口脚本组织实验，分别用于单次运行、批量对比和正式实验编排。

#### 1. 单次运行入口

单次运行入口为：

```bash
python main.py --sample 1 --run-time 60 --schedule path --quiet
```

常用参数说明如下：

| 参数 | 含义 |
| --- | --- |
| `--sample` | 选择被测样例，取值为 `1~4` |
| `--run-time` | fuzz 运行时长，单位为秒 |
| `--schedule` | 调度策略名称，如 `uniform`、`path`、`rare` |
| `--output-dir` | 结果输出根目录 |
| `--max-input-length` | 控制变异后输入长度上限 |
| `--quiet` | 关闭运行中的状态表打印 |
| `--save-population` | 额外保存最终种群快照 |

运行完成后，结果默认保存到：

```text
_result/single/
```

输出文件一般包括：

- `sample-X-schedule.pkl`
- `sample-X-schedule.json`
- `sample-X-schedule.txt`
- `sample-X-schedule-population.json`（可选）

#### 2. 批量对比入口

批量对比入口为：

```bash
python compare_schedules.py --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3
```

该脚本适合用于比较多个调度策略在不同样例上的平均效果，输出目录默认为：

```text
_result/comparison/
```

该目录下通常包括：

- `raw_results.json`
- `raw_results.csv`
- `summary.csv`
- `summary.txt`

#### 3. 正式实验编排入口

为了统一最终实验过程，本项目新增了实验套件脚本：

```bash
python run_experiment_suite.py --label official --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3 --capture-single-runs --save-population
```

该脚本会自动：

- 保存实验计划
- 保存实验矩阵
- 运行批量对比
- 可选保存代表性单次运行结果
- 自动生成报告素材汇总

输出目录为：

```text
_result/runs/official/
```

#### 4. 结果查看入口

结果查看可使用：

```bash
python view_result.py _result/single/sample-1-path.json
```

或：

```bash
python view_result.py _result/single/sample-1-path.pkl
```

此外，汇总结果可通过：

```bash
python summarize_results.py --input-dir _result
```

自动生成 `report_assets/` 下的报告素材。

## 五、六、总结与反思模板

### 六、总结与反思

通过本次实验，我们对灰盒模糊测试的基本流程形成了较完整的理解，包括语料初始化、seed 选择、输入变异、覆盖率反馈、路径调度与结果持久化等关键环节。

从实现难点来看，较有挑战性的部分主要包括：

- 如何设计有效的变异策略，使生成输入既具有随机性又具备一定针对性；
- 如何根据路径频率或覆盖率信息调整 seed 的优先级；
- 如何将单次运行、批量实验和结果汇总组织成统一、可复现的实验闭环。

从最终实验组织角度看，本组在后期进一步补充了统一的实验入口、结果持久化与汇总机制，使项目不仅能够运行，还能够较方便地支撑测试分析和实验报告撰写。

后续仍有进一步改进空间，例如：

- 增加更多面向结构化输入的 mutation 策略；
- 引入更稳定的路径标识与更细粒度的反馈信息；
- 增强中断恢复和长期运行时的中间状态管理能力；
- 增加更直观的结果可视化，例如折线图或柱状图。

总体而言，本实验帮助我们从工程实现和实验组织两个层面理解了模糊测试框架的完整工作方式。

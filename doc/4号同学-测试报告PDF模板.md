# 4号同学-测试报告PDF模板

## 使用说明

这份模板适合整理成单独的“测试报告 PDF”，内容重点不是讲实现原理，而是讲：

- 你们在什么环境下跑
- 你们怎么跑
- 你们跑出了什么
- 结果文件和截图在哪里

建议控制在 `3~6` 页之内，避免写成和实验报告重复的大篇理论说明。

---

`PJ2 模糊测试项目测试报告`

- 课程名称：软件质量保障与测试
- 实验名称：PJ2 - 模糊测试（Fuzzing）
- 小组编号：`________`
- 小组成员：`________`
- 完成日期：`________`

## 第四部分：测试内容与命令

这一部分建议拆成 4 小节。

### 4.1 单元测试

测试命令：

```bash
python -m unittest discover -s simple_fuzzer/tests -p "test_*.py"
```

测试目的：

- 验证变异器功能是否正常
- 验证路径调度与新增调度策略是否正常
- 验证实验输出与汇总脚本是否正常

测试结果填写模板：

```text
Ran ______ tests in ______s

OK
```

### 4.2 单次运行测试

测试命令示例：

```bash
python main.py --sample 1 --run-time 60 --schedule path --quiet --save-population
```

测试目的：

- 验证项目能否完成一次完整 fuzzing 流程
- 验证结果文件能否落盘
- 验证 `population` 快照能否按需保存

预期输出：

- `_result/single/*.pkl`
- `_result/single/*.json`
- `_result/single/*.txt`
- `_result/single/*-population.json`（可选）

### 4.3 批量对比测试

测试命令示例：

```bash
python compare_schedules.py --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3
```

测试目的：

- 验证多个样例和多个调度策略能否统一运行
- 验证原始结果与汇总结果是否能正确生成

预期输出：

- `_result/comparison/raw_results.json`
- `_result/comparison/raw_results.csv`
- `_result/comparison/summary.csv`
- `_result/comparison/summary.txt`

### 4.4 正式实验套件测试

测试命令示例：

```bash
python run_experiment_suite.py --label official --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3 --capture-single-runs --save-population
```

测试目的：

- 验证正式实验计划能否被完整记录
- 验证实验矩阵能否按统一参数执行
- 验证实验目录是否自动生成
- 验证报告素材是否能自动汇总

预期输出：

```text
_result/runs/official/
├─ experiment_plan.json
├─ comparison/
├─ single/
└─ report_assets/
```

## 最后建议

整理测试报告 PDF 时，推荐顺序如下：

1. 封面
2. 测试目标
3. 测试环境
4. 测试内容与命令
5. 测试结果
6. 关键截图
7. 结论

如果你们时间比较紧，这份 PDF 完全可以按这个顺序直接整理，不需要再大幅改结构。

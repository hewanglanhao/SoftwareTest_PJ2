## 新增调度策略

本部分新增 3 种互补策略，并与已有的 `uniform` 和 `path` 一起比较。

### 总体设计思路

新增调度策略的核心目标不是简单堆叠多个相似算法，而是从不同角度回答同一个问题：当前种群中“哪个 seed 更值得继续变异”。因此本部分选择了 3 个互补维度：

1. 覆盖广度：已经覆盖更多代码位置的 seed，可能更接近复杂逻辑或深层分支。
2. 覆盖稀有度：覆盖低频代码位置的 seed，可能代表尚未充分探索的区域。
3. 输入执行成本：更短的 seed 通常执行更快、变异成本更低，适合提高单位时间探索数量。

这 3 种策略分别对应 `coverage`、`rare`、`length`。它们和已有 `path` 策略形成互补：`path` 关注整条执行路径的出现频率，`rare` 关注单个覆盖位置的出现频率，`coverage` 关注覆盖规模，`length` 关注输入成本。通过这样的组合，实验对比不只是在比较“谁分数更高”，也能比较不同调度思想在不同目标程序上的取舍。

### 通用实现方式

所有新增策略都继承自 `PowerSchedule`，并复用父类的 `choose()` 逻辑。也就是说，每种策略只需要实现自己的 `assign_energy()` 方法，为种群中的每个 seed 设置 `seed.energy`；之后父类会自动根据 energy 归一化，并使用加权随机选择 seed。

这种设计保持了调度接口统一：

```text
assign_energy(population) -> 为每个 seed 计算 energy
choose(population) -> 根据 energy 加权随机选择 seed
```

其中 `coverage` 和 `length` 只依赖 seed 自身属性，因此实现为无状态调度；`rare` 需要统计运行过程中各覆盖位置出现的频率，因此额外实现 `record_path()`，由 `PathGreyBoxFuzzer.run()` 在每次执行后调用。

| 策略名 | 类名 | 设计目标 |
| --- | --- | --- |
| `uniform` | `PowerSchedule` | 基线策略，所有 seed 等概率选择。 |
| `path` | `PathPowerSchedule` | 已有策略，优先探索低频路径。 |
| `coverage` | `CoveragePowerSchedule` | 优先选择覆盖代码位置更多的 seed。 |
| `rare` | `RareCoverageSchedule` | 优先选择覆盖稀有代码位置的 seed。 |
| `length` | `LengthPowerSchedule` | 优先选择短输入，提高单位时间执行次数。 |

### 覆盖率优先调度

`CoveragePowerSchedule` 根据 seed 覆盖的位置数量分配能量：

```text
energy = 1 + len(seed.coverage)
```

该策略适合验证“已经触达更多分支的 seed，是否更值得继续变异”。

实现时，调度器遍历当前 `population`，读取每个 `Seed` 对象中的 `coverage` 集合。`coverage` 中保存的是该 seed 执行时覆盖到的代码位置，因此 `len(seed.coverage)` 可以近似表示这个 seed 的覆盖广度。覆盖位置越多，说明该 seed 触发的程序逻辑越丰富，调度器就给它更高的 energy，使其在后续 `choose()` 中更容易被选中。

代码实现保持非常直接：

```python
class CoveragePowerSchedule(PowerSchedule):
    def assign_energy(self, population):
        for seed in population:
            seed.energy = 1.0 + len(seed.coverage)
```

这里加上 `1.0` 是为了保证所有 seed 都有正 energy，即使某个 seed 的覆盖集合为空，也不会导致归一化时出现 0 权重问题。

### 稀有覆盖调度

`RareCoverageSchedule` 统计每个覆盖位置在 fuzzing 过程中出现的频率，并给覆盖稀有位置的 seed 更高能量：

```text
energy = sum(1 / location_frequency[location])
```

该策略和路径频率调度不同：路径频率关注整条路径是否常见，稀有覆盖调度关注单个代码位置是否少见。

实现时，调度器维护一个 `location_frequency` 字典：

```text
key = 覆盖位置，例如 ("sample2", 18)
value = 该位置在所有执行中出现过的次数
```

每次 fuzzer 执行完目标函数后，会把本次运行得到的 coverage 传给 `record_path()`。虽然函数名沿用了路径调度中的接口名称，但这里记录的不是整条路径 hash，而是 coverage 集合中的每个代码位置。出现次数越少的位置，代表越稀有；如果某个 seed 覆盖了多个稀有位置，它的总 energy 就会更高。

核心实现逻辑如下：

```python
class RareCoverageSchedule(PowerSchedule):
    def __init__(self):
        self.location_frequency = {}

    def record_path(self, coverage):
        for location in coverage:
            self.location_frequency[location] = self.location_frequency.get(location, 0) + 1

    def assign_energy(self, population):
        for seed in population:
            rarity = sum(
                1.0 / self.location_frequency.get(location, 1)
                for location in seed.coverage
            )
            seed.energy = max(rarity, 0.01)
```

其中 `1.0 / frequency` 体现了“越少见越重要”的思想；`max(rarity, 0.01)` 用于保证 energy 始终为正数。相比 `coverage` 策略只看覆盖数量，`rare` 策略更关注覆盖质量：覆盖很多常见位置不一定得分高，覆盖少量稀有位置也可能获得较高优先级。

### 短输入优先调度

`LengthPowerSchedule` 给短输入更高能量：

```text
energy = 1 / sqrt(len(seed.data) + 1)
```

该策略用于观察“更短、更快执行的 seed 是否能带来更高吞吐量”。

实现时，调度器读取 `seed.data` 的长度，并使用长度的平方根作为惩罚项：

```python
class LengthPowerSchedule(PowerSchedule):
    def assign_energy(self, population):
        for seed in population:
            seed.energy = 1.0 / math.sqrt(len(seed.data) + 1)
```

这里没有直接使用 `1 / len(seed.data)`，而是使用 `1 / sqrt(len(seed.data) + 1)`，原因是直接按长度反比会让长输入权重下降过快，可能导致结构复杂但有价值的长 seed 几乎没有机会被选中。平方根惩罚更温和：短输入仍然更容易被选择，但长输入不会完全被排除。

该策略本身不直接利用覆盖反馈，而是从执行效率角度优化 fuzzing。对于 `sample4` 这类 HTML 解析目标，长输入可能导致单次执行成本明显升高；短输入优先可以在同样时间内完成更多次执行，从而提高路径探索数量。

## 实验结果对比分析

本实验主要观察 4 个指标：`avg_execs` 表示单位时间内完成的执行次数，反映调度策略的运行吞吐量；`avg_paths` 表示发现的唯一路径数量，反映路径探索能力；`avg_covered_lines` 表示覆盖到的代码位置数量，是衡量覆盖率提升的核心指标；`avg_crashes` 表示发现的唯一崩溃数量，反映缺陷触发能力。由于 fuzzing 具有随机性，本次实验对每组配置重复 3 次并取平均值，以减少单次运行偶然性的影响。

从整体结果看，`sample1`、`sample2`、`sample3` 的覆盖空间相对较小，各调度策略在 1 秒内基本都能达到相同或接近相同的覆盖行数。`sample1` 中所有策略都达到 8 行覆盖和 5 条路径；`sample2` 中所有策略都达到 13 行覆盖、5 条路径和 4 个崩溃；`sample3` 中所有策略也都达到 9 行覆盖。这个现象说明对于分支结构较短、输入格式较简单的目标程序，变异器本身已经能较快触达主要分支，调度策略带来的覆盖率差异并不明显。

在这些较简单的样例中，策略差异主要体现在执行次数和崩溃发现稳定性上。以 `sample2` 为例，`coverage` 平均执行 2762.333 次，`path` 平均执行 2755 次，明显高于 `uniform` 的 2093.667 次和 `rare` 的 2116 次，但最终覆盖行数和崩溃数相同。这说明在简单样例中，更高吞吐量不一定继续转化为更高覆盖率，因为目标程序的可达路径较快被探索完毕。此时调度策略的收益主要表现为更快达到同等结果，而不是发现更多结果。

`sample4` 的结果更能体现调度策略差异。该样例使用 `HTMLParser`，输入结构更复杂，覆盖空间也更大。`coverage` 策略平均覆盖 444.333 行，明显高于其他策略，是覆盖行数表现最好的策略；但它的平均执行次数只有 232 次，平均路径数为 74.333。相反，`length` 策略平均执行 557.667 次，平均发现 177 条路径，是吞吐量和路径数量表现最好的策略，但平均覆盖行数只有 221 行。这个对比说明：覆盖率优先策略更倾向于继续探索“已经进入复杂解析逻辑”的 seed，因此能触达更多代码；短输入优先策略能更快执行更多输入，发现更多路径变体，但这些路径不一定深入到更多代码区域。

已有的 `path` 策略和新增的 `rare` 策略都属于反馈导向调度。`path` 根据整条覆盖路径的出现频率分配能量，`rare` 根据单个覆盖位置的稀有程度分配能量。在 `sample4` 中，`path` 平均覆盖 221 行、82 条路径，`rare` 平均覆盖 220.667 行、76.333 条路径，两者结果接近，说明它们确实在进行低频区域探索，但在短时间实验中没有超过 `coverage` 和 `length`。可能原因是 1 秒运行时间较短，低频反馈统计还没有充分积累；同时 `sample4` 的解析器内部路径很多，单次深入解析的收益可能比单纯偏向低频路径更明显。

从策略特性来看，`uniform` 作为基线策略实现简单，但缺少反馈信息，整体表现较稳定但没有突出优势。`coverage` 在复杂目标上覆盖行数优势明显，适合作为提升代码覆盖率的主要策略。`length` 在执行次数和路径数量上优势明显，适合需要高吞吐量、快速探索大量输入变体的场景。`path` 和 `rare` 更能体现灰盒 fuzzing 的思想：根据运行反馈动态调整 seed 的选择概率，虽然本次短实验中优势不总是最大，但它们比纯随机调度更有解释性，也更适合在长时间 fuzzing 中继续观察。

## 实验结论

本次实验表明，不同调度策略没有绝对最优，策略效果与目标程序结构和评价指标密切相关。对于 `sample1`、`sample2`、`sample3` 这类分支空间较小的目标，各策略很快达到相近覆盖率，说明变异策略和初始种子已经足以覆盖主要路径，调度策略的影响有限。

对于 `sample4` 这类复杂解析程序，调度策略差异明显。`coverage` 策略在覆盖行数上表现最好，说明优先选择高覆盖 seed 有助于深入复杂解析逻辑；`length` 策略在执行次数和路径数量上表现最好，说明短输入优先能提升 fuzzing 吞吐量和路径多样性；`path` 与 `rare` 策略虽然在本次短时间实验中没有取得最高覆盖，但它们利用路径频率和稀有覆盖信息进行反馈调度，比 `uniform` 更符合灰盒 fuzzing 的设计思想。

因此，本项目中推荐将 `uniform` 作为基线，将 `path` 作为已有核心策略，将 `coverage`、`rare`、`length` 作为新增策略进行横向比较。报告中可以重点说明：如果评价目标是覆盖行数，`coverage` 策略在当前实验中更优；如果评价目标是执行效率和路径数量，`length` 策略更优；如果评价目标是体现反馈导向调度机制，`path` 和 `rare` 更有代表性。后续如果延长运行时间或增加重复次数，`path` 和 `rare` 的低频探索优势可能会更充分地体现出来。

需要注意的是，本次实验每组运行时间为 1 秒，主要用于快速验证和横向比较，结果会受到随机种子、运行环境和样例复杂度影响。正式报告中可以将该实验作为初步对比结果，也可以在时间允许时将 `run_time` 提高到 10 秒或 30 秒，以获得更稳定的长期趋势。

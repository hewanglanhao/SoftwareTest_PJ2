## 二、实验目的

本实验旨在基于给定的简易 fuzzing 框架，完成输入变异、seed 调度、覆盖率反馈与结果持久化等关键能力，理解灰盒模糊测试的基本流程，并在此基础上形成一个可重复运行、可观察、可分析的实验闭环。

## 三、项目理解

### 3.1 框架结构

项目主要由以下模块组成：

- `fuzzer`：控制 fuzz 循环、seed 选择与输入生成。
- `runner`：执行目标样例并返回运行结果与覆盖率。
- `schedule`：对不同 seed 进行能量分配或优先级判断。
- `utils`：提供覆盖率追踪、种子封装、对象持久化与输入变异工具。
- `samples`：提供用于验证 fuzz 效果的示例程序。

### 3.2 设计主线

## 四、实现内容

### 4.1 变异器实现

本项目在 `simple_fuzzer/utils/mutator.py` 中实现并补强了变异器。`Mutator` 类维护一个变异算子列表，每次调用 `mutate()` 时随机选择一种策略，对当前 seed 生成新的候选输入。相比只做简单字符插入，当前实现同时包含字符级、字节级、块级、字典引导和 havoc 随机扰动，能够覆盖更多输入形态。

已实现的主要变异策略如下：

| 策略                    | 函数                                | 说明                                                                                 |
| ----------------------- | ----------------------------------- | ------------------------------------------------------------------------------------ |
| 随机字符插入            | `insert_random_character`         | 在任意位置插入可打印字符，提高输入长度和字符组合多样性。                             |
| 随机字符删除            | `delete_random_character`         | 删除单个字符，触发长度不足或字段缺失类边界。                                         |
| 随机块删除              | `delete_random_block`             | 删除一段短文本，模拟结构损坏、标签缺失等情况。                                       |
| bit flip                | `flip_random_bits`                | 翻转 1、2、4 个相邻 bit，模拟底层字节扰动。                                          |
| 字节算术变异            | `arithmetic_random_bytes`         | 对 1、2、4 个相邻字节加减小范围随机值，探索数值边界。                                |
| interesting values 替换 | `interesting_random_bytes`        | 将局部字节替换为 `0x00`、`0x7f`、`0xff`、最大/最小整数等边界值。               |
| 字典 token 覆盖/插入    | `overwrite_with_dictionary_token` | 注入 `FDU`、`FDUPLAB`、`<html>`、`{Key}`、`NaN` 等与样例分支相关的 token。 |
| havoc 随机插入          | `havoc_random_insert`             | 插入原输入片段或随机可打印字节，扩大随机搜索空间。                                   |
| havoc 随机替换          | `havoc_random_replace`            | 将一段内容替换为原输入片段或随机字节。                                               |
| 相邻块交换              | `random_block_swap`               | 交换两个相邻字节块，在保留局部结构的同时改变顺序。                                   |

边界情况处理方面，空字符串在无法删除或翻转时会直接返回原字符串，而插入类和字典类仍能生成非空输入；短字符串在进行 2 字节或 4 字节操作时会自动降级，避免越界；字节级变异统一先编码为 `bytearray`，变异后用 `latin-1` 解码，保证任意字节都能映射回 Python 字符串，不会因为 UTF-8 解码错误丢失变异结果；`Mutator.mutate()` 还会对非字符串输入先做 `str()` 转换，提升接口容错性。

字典 token 根据四个样例程序设计：`sample1` 对应 `0`、`1`、`-1`、`NaN`、`inf` 等数值输入；`sample2` 对应 `.`, `%d`, `{Key}` 等格式化输入；`sample3` 对应 `FDU`、`FDUPLAB`、`L`、`LA`、`LAB` 等深层分支关键字；`sample4` 对应 `<html>`、`</html>`、`<script>`、`&amp;` 等 HTML 片段。因此该变异器既保留了随机探索能力，也加入了少量面向目标程序结构的引导。

### 4.3 持久化与框架优化

为满足实验要求，我们对 seed、中间结果和最终结果的持久化方式进行了统一整理。项目通过 **simple_fuzzer/utils/object_utils.py** 中的工具函数对对象进行序列化保存，其中 **dump_object()** 和 **load_object()** 基于 **pickle** 保存和读取 Python 对象，适合保存单次运行结果；同时增加了 **dump_json()**，用于将结果保存为 **.json** 文件，方便后续统计、汇总和写入报告。

在结果落盘方面，单次运行会输出 **.pkl**、**.json** 和 **.txt** 三类结果文件；如果启用 **--save-population**，还会额外保存最终 **population** 快照。批量实验会保存原始结果和汇总结果，便于后续比较不同调度策略的表现。

在目录管理方面，持久化函数会在写文件前自动创建父目录，避免因目录不存在导致运行失败。当前项目统一使用 **_result/single/**、**_result/comparison/**、**_result/report_assets/** 和 **_result/runs/`<label>`/** 等目录保存不同类型的实验结果，使输出结构更清晰，也便于后续整理报告材料。

在长时间运行支持方面，我们采用“及时落盘关键结果”的方式减轻内存压力，将单次结果、批量结果和种群快照保存到文件系统中，而不是仅保留在内存里。同时通过 **--quiet** 参数减少无意义的状态输出，只保留关键统计信息，从而降低日志噪声，方便后续查看和截图。

### 4.4 入口与运行方式

本项目提供了单次运行、批量对比和正式实验编排三类入口，便于在不同场景下复现实验。

单次运行入口为 **main.py**，用于对某一个样例执行一次完整的 fuzzing 流程。可以通过 **--sample** 选择样例，通过 **--run-time** 设置运行时间，通过 **--schedule** 指定调度策略，还可以使用 **--quiet** 关闭状态打印，使用 **--save-population** 保存最终种群快照。例如：

`python main.py --sample 1 --run-time 60 --schedule path --quiet --save-population`

运行结果默认保存到 **_result/single/** 目录下。

批量对比入口为 **compare_schedules.py**，用于对多个样例和多个调度策略进行重复实验，并自动输出原始结果和汇总结果。例如：

`python compare_schedules.py --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3`

结果默认保存到 **_result/comparison/** 目录下。

为了方便最终实验整理，项目还提供了正式实验编排脚本 **run_experiment_suite.py**，可以自动记录实验计划、运行实验矩阵、保存代表性单次运行结果，并汇总报告素材。例如：

`python run_experiment_suite.py --label official --samples 1 2 3 4 --schedules uniform path rare --run-time 60 --repeats 3 --capture-single-runs --save-population`

结果会保存到 **_result/runs/official/** 目录下，其中包含实验计划、单次结果、批量结果和报告素材。

此外，**view_result.py** 可用于查看单次运行结果，**summarize_results.py** 可用于对结果目录进行统一汇总。通过这些入口，项目形成了从运行、对比到汇总的完整实验流程。

从结果来看，`sample1`、`sample2` 和 `sample3` 上三种策略在覆盖行数和路径数上的差异较小，说明这些样例规模相对有限，在 10 秒运行时间内各策略都能较快达到相近覆盖。`sample3` 上 `rare` 和 `uniform` 的平均崩溃数略高于 `path`，说明新增策略在该样例上的异常触发能力略强。

总体来看，本项目已经能够稳定完成 fuzzing、调度比较、结果持久化和汇总分析的完整流程。实验结果表明，不同调度策略在不同样例上的优势并不完全一致，调度效果与样例结构、输入特征和运行时间都有较大关系。

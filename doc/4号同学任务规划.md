# 4号同学任务规划

## 一、任务定位

结合 `doc/分工.md`、`doc/PJ2 - 模糊测试（Fuzzing）实验说明.md`、`doc/PJ2 - 模糊测试（Fuzzing）实验报告思路.md`，4 号同学的职责不应只理解为“写一个持久化函数”，而应理解为：

负责第 4 项编码任务的落地，并承担整个项目的实验编排、结果汇总、联调整合与报告主稿组织工作，把项目整理成一个“能稳定跑、能落结果、能直接支撑实验报告和测试报告 PDF”的最终交付版本。

## 二、依据文档提炼出的核心要求

### 1. 来自实验说明的要求

`PJ2 - 模糊测试（Fuzzing）实验说明.md` 中明确提出了以下与 4 号同学直接相关的内容：

- 需要使用 `object_utils` 将 `seed` 或相关中间结果持久化到文件系统，避免长时间运行时内存持续增长。
- 最终提交物不只有代码，还包括：
  - 代码实现
  - 实验报告
  - 测试报告 PDF
- 测试报告 PDF 建议包含：
  - 运行环境说明
  - 目标样例与运行参数
  - 覆盖率变化或崩溃统计结果
  - 关键实现截图或日志片段

这意味着 4 号同学除了负责持久化功能，还要负责把实验过程整理成可复现、可展示、可写报告的标准结果。

### 2. 来自实验报告思路的要求

`PJ2 - 模糊测试（Fuzzing）实验报告思路.md` 中，4 号同学最需要负责或参与撰写的部分主要有：

- `4.3 持久化与框架优化`
- `4.4 入口与运行方式`
- `五、测试与结果`
- `六、总结与反思` 中与整体实验组织、结果分析相关的部分

报告思路还特别强调了几个点：

- 项目要形成“可重复运行、可观察、可分析的实验闭环”。
- 需要说明如何处理 `seed` 持久化、结果落盘和长时间运行时的内存控制问题。
- 需要明确实验入口如何运行、如何切换样例、如何设置 fuzz 时间，以及结果保存位置。
- 测试与结果部分需要准备样例编号、运行时间、初始种子数量、覆盖率变化、崩溃数、典型输出截图等内容。

因此，4 号同学的工作重点应落在“交付层”和“实验组织层”，而不只是算法层。

## 三、当前项目现状

从当前仓库来看，4 号同学相关的基础设施已经有一部分实现：

- `simple_fuzzer/utils/object_utils.py`
  - 已提供 `dump_object()`、`load_object()`、`get_md5_of_object()`。
  - 已具备基础对象序列化与自动创建目录能力。
- `simple_fuzzer/main.py`
  - 已支持选择 `sample`、设置 `run-time`、指定 `output-dir`、选择 `schedule` 等参数。
  - 已将单次运行结果持久化为 `_result` 下的 `Sample-X.pkl`。
- `simple_fuzzer/utils/result.py`
  - 已有 `Result` 对象，能够统一封装覆盖行、崩溃、运行时长等结果。
- `simple_fuzzer/view_result.py`
  - 已支持将持久化结果转为文本或 JSON 查看。
- `simple_fuzzer/compare_schedules.py`
  - 已支持批量运行多个 `sample`、多个 `schedule`，并导出 `raw_results.json`、`raw_results.csv`、`summary.csv`、`summary.txt`。

这说明 4 号同学的工作不是从零开始，而是在已有基础上做统一、补强和交付整理。

## 四、4号同学的更新后职责

### 1. 编码职责

围绕实验说明中的第 4 项任务，4 号同学应重点负责：

- 使用 `object_utils` 进一步完善持久化逻辑。
- 明确哪些对象需要落盘：
  - 最终运行结果
  - 可选的中间统计结果
  - 可选的 `seed` 或 `population` 快照
- 规范输出目录和输出文件格式，避免结果分散、命名混乱。
- 统一运行参数和实验入口，保证结果可复现。
- 减少不必要的调试输出，保留关键统计信息。

### 2. 联调职责

4 号同学需要承担整个项目的联调整合工作，包括：

- 合并 1、2、3 号同学的代码后检查是否能统一运行。
- 确认所有 `schedule` 都能从统一入口调用。
- 确认不同样例在统一参数下都可以完成 fuzzing。
- 确认结果输出字段一致，便于后续汇总。
- 协调实验参数口径，避免每个人各跑各的、结果无法横向比较。

### 3. 实验职责

4 号同学是实验组织和结果汇总的负责人，建议承担：

- 统一设计实验矩阵。
- 组织运行 `sample1 ~ sample4` 的整体实验。
- 收集覆盖率、路径数、崩溃数、执行次数、运行时间等指标。
- 输出表格、文本摘要、截图和日志片段。
- 整理可直接用于报告与测试 PDF 的结果材料。

### 4. 文档职责

4 号同学建议主写或统稿以下内容：

- 小组分工说明
- 持久化与框架优化
- 入口与运行方式
- 测试与结果
- 整体实验总结
- 测试报告 PDF 中的运行说明和结果整理部分

## 五、推荐实施方案

### 第一阶段：统一实验接口与输出格式

目标是先把“实验怎么跑、结果怎么存”统一下来。

建议完成：

- 统一单次运行和批量运行的参数风格。
- 统一结果字段，建议至少包括：
  - `sample`
  - `schedule`
  - `run_time`
  - `initial_seed_count`
  - `covered_line_count`
  - `crash_count`
  - `total_paths`
  - `total_execs`
  - `duration_seconds`
- 统一输出目录，例如：
  - `_result/single/`
  - `_result/comparison/`
  - `_result/report_assets/`

这一阶段的目标不是做复杂优化，而是把结果出口标准化。

### 第二阶段：补强持久化与汇总能力

在已有 `pickle` 持久化基础上，建议继续补：

- 中间结果持久化
  - 例如按阶段保存 `seed`、`population`、统计结果
- 结果汇总脚本
  - 将多个实验输出整理为统一的 `csv/json/txt/md`
- 结果查看脚本增强
  - 便于快速导出报告所需的数据片段

如果时间有限，优先保证“最终结果统一落盘 + 自动汇总成表”。

### 第三阶段：组织正式实验

根据报告要求，建议统一实验方案：

- 对 `sample1 ~ sample4` 至少各跑一组实验。
- 对主要调度策略至少做一轮对比：
  - `uniform`
  - `path`
  - 新增策略
- 如果时间允许，每组做 `repeats=3`，取平均值，结果更稳定。
- 固定随机种子、运行时长、最大输入长度等参数，确保可复现。

建议提前约定实验记录表头，避免后续再人工补数据。

### 第四阶段：整理报告与测试 PDF 材料

4 号同学需要边跑实验边积累材料，不要等代码全部完成后再回头补。

建议同步保存：

- 运行命令
- 运行参数
- 结果表格
- 关键日志片段
- 终端输出截图
- 典型样例说明

这样最后写实验报告和测试报告 PDF 时会非常省力。

## 六、建议输出的最终成果

4 号同学最终最好整理出以下成果：

### 1. 代码成果

- 稳定可运行的最终版本
- 统一实验入口
- 结果持久化与汇总脚本
- 清晰的输出目录结构

### 2. 数据成果

- 单次运行结果文件
- 多策略对比结果文件
- 汇总表格
- 可直接引用到报告中的统计数据

### 3. 文档成果

- 实验报告主稿相关章节
- 测试报告 PDF 素材
- 运行命令说明
- 最终结果摘要表

## 七、建议的实验表格字段

为了后续报告和测试 PDF 统一，建议最终汇总表至少包含以下字段：

| 字段 | 含义 |
| --- | --- |
| `sample` | 样例编号 |
| `schedule` | 调度策略名称 |
| `run_time` | 设定运行时间 |
| `initial_seed_count` | 初始种子数量 |
| `covered_line_count` | 覆盖行数 |
| `crash_count` | 崩溃或异常数量 |
| `total_paths` | 探索到的唯一路径数 |
| `total_execs` | 总执行次数 |
| `duration_seconds` | 实际运行时长 |

如果后续还想做更细致分析，也可以增加：

- `seed`
- `repeat`
- `output_file`
- `notes`

## 八、建议时间安排

### 第 1 步：先统一口径

- 确认所有人输出结果的字段和目录结构。
- 确认最终实验使用哪些 `schedule`。
- 确认最终使用哪些样例和运行参数。

### 第 2 步：补实验支撑脚本

- 完成结果汇总脚本。
- 检查单次运行和批量运行脚本是否稳定。
- 确认查看结果的方式足够方便。

### 第 3 步：正式跑实验

- 统一参数跑完整实验。
- 保存结果文件、截图和关键日志。
- 对结果进行初步核对，避免漏数据。

### 第 4 步：整理文档

- 输出结果总表。
- 撰写实验报告相关章节。
- 整理测试报告 PDF 所需内容。

## 九、结论

综合项目现状和课程文档要求，4 号同学的角色应定义为：

项目最终交付整合负责人。

其任务重点不是继续扩展某一种调度算法，而是围绕“持久化、统一入口、实验运行、结果汇总、报告支撑”这条主线，把现有代码整理成一个可交付、可复现、可展示的完整实验作品。

如果执行顺利，4 号同学最终应能交出三类关键成果：

- 一个统一、稳定、可运行的项目版本
- 一套可直接用于报告的实验结果与统计表
- 一份能支撑实验报告和测试报告 PDF 的完整材料集合
